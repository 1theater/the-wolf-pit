# «Волчья яма» — сайт-читалка на GitHub Pages — дизайн

> Статический сайт, который рендерит markdown-файлы проекта в читаемые веб-страницы.
> Хостинг — GitHub Pages (репозиторий `1theater/the-wolf-pit`).

---

## 1. Цель и критерии успеха

Сделать статический сайт, на котором можно прочитать пьесу «Волчья яма» с телефона и десктопа. В начале страницы пьесы — постер. Сайт многостраничный (пьеса + книга ролей, в будущем — сцены, свет) с простым меню. При изменении `.md`-файла и пуше изменения подтягиваются на сайт без ручной пересборки.

**Критерии успеха:**
- Сайт открывается по `https://1theater.github.io/the-wolf-pit/` и показывает пьесу с постером сверху.
- Меню переключает между «Пьеса» и «Книга ролей» без перезагрузки.
- Текст читаемо отрендерен (заголовки, реплики, ремарки, цитаты, разделители) и корректно выглядит на мобильном и десктопном экране.
- Правка `.md` + пуш → изменения видны на сайте при следующем заходе.
- Добавить новую главу = положить `.md` в корень + дописать одну строку в список страниц.

## 2. Архитектура (клиентский рендер, без сборки)

Одностраничное приложение (SPA): один `index.html` загружает CSS и JS, JS на лету читает нужный `.md` из того же репозитория и рендерит его в HTML через вшитую markdown-библиотеку. Никакого билд-пайплайна — GitHub Pages просто отдаёт статические файлы как есть.

**Почему так (а не GitHub Actions / Jekyll):** требование «правлю md → сайт обновился» выполняется сразу и без пайплайна, потому что сайт читает «живой» `.md`. Меньше движущихся частей, проще поддержка и расширение.

### Проверенные факты GitHub Pages (из официальной доки)
- **URL сайта проекта:** `http(s)://<owner>.github.io/<repositoryname>` → `https://1theater.github.io/the-wolf-pit/`. Имя репозитория может быть любым (особое имя `<owner>.github.io` нужно только для сайта пользователя/организации).
- **Источник публикации (deploy from a branch):** только корень `/` или папка `/docs`; произвольную подпапку выбрать нельзя. При публикации из корня публикуется **весь репозиторий, включая подпапки**.
- Поэтому `index.html` и `.md` лежат в одной публикуемой области (корень), и `index.html` остаётся в корне ради чистого URL.

## 3. Раскладка файлов (Вариант 1: публикация из корня)

```
index.html                 — точка входа: шапка с меню + контейнер контента
.nojekyll                  — отключает Jekyll (см. §7)
assets/
  style.css                — тёмная тема, адаптив, типографика
  app.js                   — роутер, загрузка и рендер markdown
  markdown-it.min.js       — библиотека рендера, вшита в репозиторий (без CDN)
pages.json                 — список страниц меню
Волчья яма.md              — текст пьесы (остаётся в корне)
Книга ролей.md             — книга ролей (переносится из docs/, см. §8)
poster.png                 — постер (уже в корне)
```

Спеки и планы остаются в `docs/superpowers/` — они не для читателя и не публикуются на сайте как страницы (на них нет ссылок; при публикации из корня их сырые файлы технически доступны по прямому URL, но это внутренние документы без вреда).

## 4. Список страниц — `pages.json`

```json
[
  { "slug": "play",  "title": "Пьеса",       "file": "Волчья яма.md", "poster": "poster.png" },
  { "slug": "roles", "title": "Книга ролей", "file": "Книга ролей.md" }
]
```

- `slug` — идентификатор для адреса (`#play`, `#roles`); стабилен при переименовании заголовка.
- `title` — подпись в меню.
- `file` — имя `.md` (при запросе кодируется через `encodeURIComponent` — корректно работает с кириллицей и пробелами).
- `poster` — опционально; если задан, картинка вставляется в начало этой страницы.

Добавление будущей главы (сцены, свет) = положить `.md` в корень + одна строка в этот список.

## 5. Поведение (`assets/app.js`)

Небольшие функции с одной задачей каждая:

- `loadManifest()` — `fetch('pages.json')` → массив страниц. При ошибке — показать сообщение.
- `buildNav(pages)` — построить меню из списка (ссылки вида `#slug`), пометить активный пункт.
- `resolveRoute(pages)` — взять `slug` из `location.hash`; если пусто или неизвестно — первая страница.
- `loadPage(entry)` — `fetch(encodeURIComponent(file) + '?v=' + Date.now())` → текст md. Метка `?v=` — анти-кеш (см. §6). При ошибке — `showError()`.
- `render(md, entry)` — `markdownit(...).render(md)` → HTML; если у `entry` есть `poster`, добавить `<img>` перед контентом; вставить в контейнер; проставить `document.title`; прокрутить наверх.
- `showError(msg)` — понятное сообщение в области контента вместо пустоты.

Поток: загрузить манифест → построить меню → отрисовать текущий маршрут. Слушатель `hashchange` перерисовывает страницу при переключении пункта меню.

**Настройки markdown-it:** `html: false` (сырой HTML не нужен и безопаснее выключить), `linkify: true`, `typographer: true`. Поддерживаются все конструкции пьесы: `#`/`##`/`###`, `**жирный**`, `*курсив*`, `> цитата`, `---`.

## 6. Авто-обновление и кеширование

GitHub Pages отдаёт файлы через CDN с `Cache-Control: max-age=600` (~10 минут). Чтобы правки `.md` были видны **сразу**, запрос `.md` идёт с анти-кеш меткой `?v=<timestamp>`. Платой за это является то, что `.md` не кешируется браузером, но файлы небольшие — это незаметно. `index.html`/`assets` кешируются обычным образом (это статика, меняется редко).

## 7. `.nojekyll` — почему обязателен

Без файла `.nojekyll` GitHub Pages прогоняет содержимое через Jekyll. Jekyll может вмешаться в обработку файлов (особенно с именами на кириллице/с пробелами и `.md`), из-за чего «сырой» `.md` может стать недоступен для `fetch`. Пустой файл `.nojekyll` в корне отключает Jekyll — файлы отдаются ровно как лежат. Для нашего клиентского рендера это обязательное условие.

## 8. Перенос книги ролей

`docs/superpowers/specs/2026-06-14-volchya-yama-roli.md` → `Книга ролей.md` в корне (через `git mv`, чтобы сохранить историю). Внутренние ссылки в книге ролей на другие файлы отсутствуют, правки содержимого не требуются.

## 9. Стиль (тёмная тема, адаптив)

- **Цвета** через CSS-переменные: тёмный фон (≈`#15171b`), светлый текст (≈`#e6e3dc`), приглушённый кроваво-красный акцент для заголовков и `---`. Сдержанно, под нуар-триллер.
- **Колонка чтения:** `max-width ≈ 42rem`, по центру, отступы; высота строки ≈1.7; базовый кегль ≈18px, на мобильном — адаптивно.
- **Шрифты системные** (ничего не качается): серифный для текста пьесы (`Georgia, 'Times New Roman', serif`) ради комфортного длинного чтения; системный без-серифный для меню/шапки.
- **Семантика пьесы в CSS:** `em` (ремарки) — приглушённый цвет, курсив; `strong` (реплики) — выделен; `blockquote` (радио/цитаты) — с левой линией и отступом; `hr` — тонкий разделитель акцентного цвета; `h1/h2/h3` — масштаб по уровню.
- **Постер:** `max-width: 100%`, по центру, скруглённые углы, отступ снизу.
- **Шапка/меню:** компактная панель сверху (название + ссылки), переносится в строку. Гамбургер-меню не делаем, пока пунктов мало (YAGNI).

## 10. Обработка ошибок

- `.md` не загрузился (404/сеть) → сообщение «Не удалось загрузить страницу» в области контента.
- `pages.json` не загрузился → сообщение об ошибке вместо меню.
- Неизвестный `#slug` → открыть первую страницу.

## 11. Включение GitHub Pages

Настройка делается в репозитории: **Settings → Pages → Build and deployment → Source: Deploy from a branch → Branch: `main` → Folder: `/ (root)` → Save**.

Попробуем включить программно через `gh` CLI (`gh api`); если нет прав/аутентификации — даём пользователю шаги выше. После включения сайт доступен по `https://1theater.github.io/the-wolf-pit/` (первая сборка занимает 1–2 минуты).

## 12. Проверка (перед коммитом и после деплоя)

- **Локально:** `python -m http.server` в корне, открыть `http://localhost:8000/`. (Через `file://` не открывать — `fetch` локальных файлов блокируется браузером.)
- Проверить: обе страницы рендерятся; постер виден на странице пьесы; меню переключает; все конструкции markdown отображаются.
- **Адаптив:** проверить узкий (мобильный) и широкий экран в devtools.
- **После деплоя:** открыть боевой URL, повторить проверки; проверить авто-обновление (мелкая правка `.md` + пуш → видно на сайте).

## 13. Что НЕ входит (YAGNI)

- Нет поиска, оглавления внутри страницы, переключателя тем (взята тёмная), гамбургер-меню.
- Нет сборки/CI-пайплайна (Actions/Jekyll).
- Нет внешних CDN и веб-шрифтов — всё локально и системно.
- Нет привязки к конкретному домену (используем стандартный `github.io`-адрес).
